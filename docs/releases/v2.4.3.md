# Go v2.4.3

v2.4.3 is a synchronized bot and `ledger-chain-watcher` release. It combines permission-hierarchy recovery with bounded chain-gap convergence. Back up both PostgreSQL databases, deploy the watcher first, verify readiness, and then deploy every bot instance.

## Permission hierarchy and recovery

- Normalizes global operators into explicit primary and secondary levels using host, parent, creator, and audit evidence.
- Quarantines ambiguous identities instead of silently granting broader access.
- Archives broadcast scopes for disabled identities and restores valid snapshots only when an identity is explicitly re-enabled.
- Keeps single-group ledger operators scoped to that group; they do not gain invite, admin, broadcast, or unlimited address-watch capabilities.
- Records forward-only migration gates, including `2.4.3-broadcast-permission-restore`.

Production-specific permission recovery data is not included in this repository or Release. Deployment operators must use only the separately handed-off SQL artifact after verifying its SHA256 and database preconditions.

## Chain gap convergence

- Adds bounded realtime scan overlap with persistent gap tasks and lease generations.
- Coalesces compatible historical window gaps without merging tasks that have different `head_event_id` boundaries.
- Normalizes stale or overlapping gap backlog at startup and keeps readiness degraded until required gaps are drained.
- Adds `CHAIN_WATCHER_CATCHUP_MAX_INFLIGHT=3`; catch-up remains bounded by request budgets, per-key throttling, and realtime reservations.
- Preserves priority-zero notification outbox delivery and cross-source event deduplication.

## Required rollout checks

1. Back up every bot database and the watcher database.
2. Preserve `CHAIN_WATCHER_KEY_ENCRYPTION_KEY` and stable, unique `BOT_FALLBACK_INSTANCE_ID` values.
3. Add `CHAIN_WATCHER_CATCHUP_MAX_INFLIGHT=3` to the watcher environment.
4. Deploy the v2.4.3 watcher first and verify `/healthz`, `/readyz`, and authenticated `/status`.
5. Deploy every bot with `ghcr.io/dayou0168/telegram-ledger-bot-go:2.4.3`.
6. Verify the watcher migration `2.4.3`, bot migration `2.4.3-broadcast-permission-restore`, permission snapshots, gap indexes, logs, and a real USDT transfer.

PostgreSQL migrations are forward-only. Runtime rollback does not downgrade schema. Follow [the v2.4.3 production rollout guide](../production-rollout-v2.4.3.md) and use the immutable digests and checksums from the GitHub Release.
