# Go v2.4.2

v2.4.2 is a synchronized bot and `ledger-chain-watcher` upgrade. Deploy the watcher first, verify readiness, and then deploy every bot instance in the same maintenance window.

## Highlights

- Ledger clear now uses a 60-second confirmation ticket bound to the initiating user, chat, and active ledger period. Ticket consumption and record deletion are serialized and committed atomically, so another user, an expired ticket, a changed period, or a repeated click cannot clear records.
- Global permissions use the independent `global_operators` primary/secondary model. Single-group `operators` remain local ledger permissions and do not gain invitation, admin, broadcast, or unlimited address-watch capabilities.
- Ledger-period handling consistently uses the active period key and start time. Disabled cutoff periods survive midnight, re-enabling cutoff does not truncate the current period, and undo/clear operations stay inside the current active period.
- Broadcast administration rechecks targets at send time, keeps group rename/delete permission updates transactional, preserves the lightweight bulk lane, and records private-cleanup categories without storing message bodies.
- The watcher pipeline persists realtime watermarks, fallback leases, exact gap tasks, minute metrics, and source event identity. Main scans allow at most three bounded in-flight rounds; failed pages and missing anchors become fenced gap work instead of silently advancing the cursor.
- `/healthz` remains liveness only. `/readyz` reflects source freshness, cursor continuity, and open gaps. Authenticated `/status` exposes bounded diagnostics without returning API keys, secrets, or raw anchors.

## Required configuration

Before starting v2.4.2, verify these values through the existing secure deployment channel:

- `CHAIN_WATCHER_KEY_ENCRYPTION_KEY`: the existing base64-encoded 32-byte AES key used for the watcher key registry. Losing or replacing it makes stored keys unreadable.
- `CHAIN_WATCHER_ADMIN_TOKEN`: a separate random token protecting `/status`.
- `CHAIN_WATCHER_MAIN_SCAN_TIMEOUT_MS=3000`.
- `CHAIN_WATCHER_MAIN_MAX_INFLIGHT_ROUNDS=3`.
- `BOT_FALLBACK_SHARED_DATABASE_URL`: each bot points this to the shared watcher PostgreSQL database.
- `BOT_FALLBACK_INSTANCE_ID`: unique, stable, and different for every bot instance.

Do not restore the removed per-bot emergency scanner settings or configure `CHAIN_WATCHER_MAIN_SCAN_MAX_INFLIGHT`; they are not part of the v2.4.2 runtime.

## Database and deployment warning

Back up every bot PostgreSQL database and the watcher PostgreSQL database before upgrading. The application performs forward-only schema migrations, including active-period metadata, clear tickets, global-operator safety migrations, chain event identity, watcher gap tasks, fallback lease state, and metric indexes. Rolling an executable or image back does not roll the schema back.

Upgrade in this order:

1. Install the v2.4.2 watcher binary or image.
2. Confirm `/healthz` is healthy, `/readyz` is ready, and authenticated `/status` shows bounded in-flight rounds with no unresolved migration error.
3. Deploy all bot instances with the v2.4.2 image.
4. Confirm bot migrations, watcher event claims, priority-0 notification outbox delivery, and Telegram send logs.
5. Perform a real USDT incoming and outgoing transfer test; health endpoints do not replace this end-to-end check.

Use `docs/production-rollout-v2.4.2.md` and `deploy/production-rollout.sh` for backup, preflight, watcher-first rollout, bot rollout, rollback limitations, and acceptance checks.

## Verification and artifacts

The explicit release workflow runs Go formatting checks, shell and YAML syntax validation, tracked-secret pattern checks, PostgreSQL 16 integration tests, `go vet`, bot/watcher builds, and race tests for critical packages before the release job can run.

The Release contains the Linux amd64 watcher binary, checksum files, a watcher installation archive, a source archive, aggregate `SHA256SUMS`, and `image-digests.txt`. Digests are generated only after GHCR publication and are not hard-coded in this document.
