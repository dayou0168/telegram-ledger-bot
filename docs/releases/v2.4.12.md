# Go v2.4.12

v2.4.12 improves operator target discovery and adds explicit per-user broadcast reply notification controls. It is a bot-side release with two forward-only PostgreSQL migrations; the watcher protocol is unchanged.

## Operator and member discovery

- Operator commands accept a replied-to user, a structured Telegram `text_mention`, or a previously recorded `@username`.
- Users without a username can be selected by replying to their message or using Telegram's clickable-name mention.
- Invalid users, bots, the command sender, and duplicate targets are ignored by UID.
- Plain typed display names and numeric UIDs are deliberately not accepted as operator targets.
- Message updates record `has_spoken=true`; `chat_member` updates refresh identity only and do not make a member eligible for mention-all broadcasts.

## Broadcast reply notifications

- The admin page has a separate “回复通知” tab with responsive desktop and mobile layouts.
- The original broadcast sender always receives replies.
- Host, default operators, and active global operators may configure additional recipients within their existing authority.
- A primary operator may manage only direct active secondary operators, which default to enabled.
- Secondary and group-only operators have no management entry; forged unauthorized POST requests return 403.
- Preference lookups are batched. If preferences cannot be read, delivery falls back to the original sender only.

## Database and compatibility

- Adds migration `2.4.16-chat-member-discovery` for `users.has_spoken` and its partial lookup index.
- Adds migration `2.4.17-broadcast-reply-preferences` for durable per-user preferences and lookup indexes.
- Both migrations are transactional, ordered after the existing 2.4.15 migration, and safe to re-open after successful application.
- No watcher protocol, scheduler, database, or chain-source behavior changes.
- Production should update bot containers only. The release workflow still publishes matching watcher artifacts, but an existing v2.4.11 watcher does not need to restart.
