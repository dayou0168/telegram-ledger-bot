# Go v2.4.7

v2.4.7 is an urgent chain-gap convergence fix for v2.4.6. It prevents progressed recovery work from rewinding, rotates the fair lane across windows, and separates realtime source health from historical continuity. Both `ledger-chain-watcher` and the bot must be upgraded together.

## Fixes

- Preserves `next_page` when overlapping gap tasks are normalized or merged. A progressed task may append one not-yet-started successor instead of being reset.
- Rotates fair claims by the last processed time and adds `idx_chain_watcher_gap_fair_claim`, preventing one old large task from starving other windows.
- Schedules catch-up work every 250ms with token awareness. A defer that makes no upstream API call is no longer counted as a failure.
- Exposes `source_ready` separately from `continuity_ready`. Historical gap backlog may report degraded continuity without causing the bot to start public fallback scanning.
- Records the forward-only migration marker `2.4.7-chain-gap-convergence` in both bot and watcher databases.

## Upgrade Order

1. Verify Release checksums and immutable image digests.
2. Back up both the bot PostgreSQL database and the separate watcher PostgreSQL database.
3. Upgrade `ledger-chain-watcher` first.
4. Require `/healthz`, authenticated `/status`, and `source_ready=true`. `continuity_ready=false` is acceptable while historical gaps converge.
5. Upgrade every bot instance to the matching v2.4.7 image.
6. Verify the new migration marker and fair-claim index, then watch gap counts decrease without page rewind or fallback activation.

Use [the v2.4.7 production rollout guide](../production-rollout-v2.4.7.md) for the exact acceptance boundary.

## Residual Acceptance

PostgreSQL integration, race, vet, and build gates cover this release. Real Tronscan throttling, existing production gap convergence, production load, and real incoming/outgoing USDT transfer timing remain deployment acceptance items.
