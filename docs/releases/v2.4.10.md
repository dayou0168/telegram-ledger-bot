# Go v2.4.10

v2.4.10 makes Telegram intake, private routing, and quick replies durable while reducing latency on the ledger hot path. It adds forward-only PostgreSQL migrations and requires the watcher and every bot instance to be upgraded together.

## Ledger Performance

- Caches coherent global permission capabilities and actively invalidates them after permission changes.
- Maintains materialized per-period ledger summaries with shadow/safe reconciliation instead of rebuilding every summary from the full record set.
- Commits ledger mutations and critical receipt outbox rows atomically, then wakes the critical sender directly.
- Isolates update admission and critical/normal/bulk send queues so broadcasts and slow work do not occupy the ledger response path.

## Durable Telegram State

- Persists Telegram updates in `telegram_update_inbox`, with leases, retries, route ordering, deduplication, and terminal cleanup.
- Persists private-chat routing in `telegram_private_route_states`, so broadcast and quick-reply state survives a process restart.
- Uses actual execution time for permission checks, the current accounting period, and clear-ticket expiry. A queued update cannot reuse an expired 60-second clear ticket or apply to a stale period.
- Invalidates clear tickets when the cutoff changes and keeps initiator and accounting-period binding intact.

## Durable Quick Replies

- Persists quick replies in `telegram_quick_reply_outbox` before the inbox update is acknowledged.
- Checks authorization both when enqueueing and immediately before delivery; revoked operators cannot drain queued replies.
- Recovers expired processing leases after a crash.
- Retries Telegram 429, 5xx, and network failures; treats 400 and 403 as terminal; removes terminal rows after 72 hours.
- Preserves the existing lightweight quick-reply UX while separating durable delivery from the inbound update lease.

Telegram may accept a request while its HTTP acknowledgement is lost. That external ACK uncertainty is the remaining duplication boundary and cannot be removed completely without a Telegram-side idempotency key.

## Database Migration

- Records `2.4.14-telegram-private-route-state` after creating the durable update inbox and private route state structures.
- Records `2.4.15-telegram-quick-reply-outbox` after creating the durable quick-reply outbox and its due, stream/actor, lease, and cleanup indexes.
- Migrations are forward-only and run from the shared storage bootstrap. Back up both the bot and watcher PostgreSQL databases before the runtime update.

## Upgrade Order

1. Verify all Release checksums and both immutable image digests.
2. Back up both PostgreSQL databases and preserve current runtime files.
3. Upgrade `ledger-chain-watcher` to v2.4.10 and require `/healthz`, authenticated `/status`, and `source_ready=true`.
4. Upgrade every bot instance to the v2.4.10 image.
5. Confirm both migration markers, all new tables/indexes, queue recovery, and Telegram delivery behavior.

Use [the v2.4.10 production rollout guide](../production-rollout-v2.4.10.md) for the exact acceptance and rollback boundaries.
