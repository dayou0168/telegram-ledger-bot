# Go v2.4.6

v2.4.6 is a chain-monitoring scheduler release. It keeps realtime P1 work responsive while turning deeper recovery into persistent, fair, resumable gap tasks. Both `ledger-chain-watcher` and the bot changed and must be upgraded together.

## Highlights

- Plans sustainable request tokens per active API key without treating the configured daily quota as a local hard stop; real upstream `429` responses remain authoritative.
- Reserves an API and persistence lane for realtime P1 scanning while bounding P2-PN persistence and catch-up concurrency.
- Converts anchor expansion and time-window recovery into persistent gap tasks with exact continuation, overlap merging, `retry_after`, bounded fairness, and aging.
- Adds dynamic catch-up workers, head scan time/page/concurrency controls, and minute-level gap scheduler metrics exposed through authenticated `/status`.
- Writes chain notifications to the bot outbox atomically at priority `0` and immediately wakes the critical Telegram send lane.

## Database Migration

The forward-only migration `2.4.6-chain-gap-scheduler` runs in both the bot and watcher databases. It adds:

- `chain_watcher_gap_tasks.retry_after`;
- `idx_chain_watcher_gap_ready_claim` and supporting gap lookup indexes;
- `chain_watcher_gap_metric_minutes` for retained scheduler metrics.

Back up both PostgreSQL databases before upgrading. Runtime rollback does not remove these schema objects.

## Upgrade Order

1. Verify all Release checksums and immutable image digests.
2. Back up the bot database and the separate watcher database.
3. Upgrade `ledger-chain-watcher` first and require `/healthz`, `/readyz`, and authenticated `/status` to pass.
4. Upgrade every bot instance to the matching v2.4.6 image.
5. Verify both databases contain the migration marker exactly once and monitor gap/outbox metrics.

Use [the v2.4.6 production rollout guide](../production-rollout-v2.4.6.md) for exact preflight and acceptance checks.

## Residual Acceptance

Local PostgreSQL integration, race, vet, and build gates cover this release. Real Tronscan traffic, real `429` recovery, production load, and real incoming/outgoing USDT transfer timing remain deployment acceptance items and must be observed after rollout.
