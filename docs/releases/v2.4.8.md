# Go v2.4.8

v2.4.8 is an urgent watcher-only patch for v2.4.7. It fixes cancellation attribution when one concurrent scan completes and cancels the shared parent round context. Production only needs the watcher update; existing v2.4.7 bot containers may remain running.

## Fixes

- Treats cancellation inherited from an already-cancelled parent scan round as orchestration cleanup, not as a Tronscan Key transport failure.
- Prevents a healthy Key from being cooled down merely because another concurrent request completed the same round first.
- Preserves the existing health policy for real HTTP timeouts, network failures, and 5xx responses.
- Adds regression coverage at the Tron client, Key pool, and watcher round levels.

## Compatibility

- No PostgreSQL migration is added.
- Bot business behavior is unchanged.
- `/healthz`, `/readyz`, `/status`, `source_ready`, `continuity_ready`, and fallback contracts are unchanged.
- The release workflow still publishes matching v2.4.8 bot and watcher images plus the complete eight-asset Release set. The matching bot image is reproducible but not required for this production hotfix.

## Upgrade Order

1. Verify Release checksums and the immutable watcher digest.
2. Preserve the current watcher binary, environment file, unit, and recent status/log evidence.
3. Upgrade only `ledger-chain-watcher` to v2.4.8 and restart it.
4. Require `/healthz`, authenticated `/status`, and `source_ready=true`.
5. Confirm parent-round cancellations no longer cool healthy Keys while genuine upstream failures still do.
6. Leave the bot on v2.4.7 unless an operator separately chooses the matching v2.4.8 image.

Use [the v2.4.8 production rollout guide](../production-rollout-v2.4.8.md) for the exact acceptance boundary.

## Residual Acceptance

PostgreSQL integration, race, vet, and build gates cover this release. Real Tronscan throttling, genuine upstream timeout behavior, production concurrency, and real incoming/outgoing USDT transfer timing remain deployment acceptance items.
