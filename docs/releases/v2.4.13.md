# Go v2.4.13

v2.4.13 hardens admin identity and adds host-controlled message observers, reliable media-aware broadcast delivery, durable broadcast targets, and an explicit disabled day-cutoff mode. It is a bot-side release with two forward-only PostgreSQL migrations; the watcher protocol is unchanged.

## Admin identity and observer authorization

- Admin login accepts only a Telegram one-time ticket. The ticket UID is authoritative, and the current host/global-operator role is reloaded for login and every authenticated request.
- A disabled global operator immediately loses an existing admin session. A group-only operator cannot enter the admin site.
- `ADMIN_SESSION_SECRET` is the independent session-signing secret and is required for production. If it is absent, session creation and verification fail closed.
- `ADMIN_WEB_TOKEN` is only an optional second factor for a valid Telegram ticket. It cannot log in by itself, assign a role, or sign a session.
- Only the host can create, update, or revoke cross-primary observer grants. Forged host UIDs with a non-host role and primary/secondary requests receive 403.

## Reliable broadcast delivery

- Secondary broadcasts and reply notices preserve text, photo, and photo-with-caption payloads in the durable notification outbox.
- Broadcast upstream copies have stable dedupe keys and retain the Telegram message mapping needed for reply threading after delivery.
- Retryable Telegram/network failures remain pending with backoff. Sent outbox rows can be cleaned without losing the durable upstream mapping, while pending rows are retained.
- The source secondary, host, direct primary, and explicitly authorized observer primary are resolved independently. Broadcast and reply observation are separate grant switches, and reply preferences remain per recipient.
- Revocation is rechecked before send and removes the saved broadcast target. Group rename/delete changes update or remove durable targets transactionally.

## Durable targets and disabled cutoff

- The selected broadcast chat or group is stored in PostgreSQL and restored after restart; it no longer expires with the 30-minute private-menu state.
- `关闭日切` and `设置日切-1` both select the disabled cutoff mode.
- An active disabled-cutoff ledger period continues across midnight and restart with its original period identity.
- A manually stopped or inactive group remains stopped; disabling cutoff never starts accounting by itself.

## Database and compatibility

- Adds migration `2.4.18-operator-message-observers-admin-identity` for observer grants/audit state and admin identity compatibility.
- Adds migration `2.4.19-broadcast-delivery-state` for durable broadcast targets, upstream message mappings, and media-aware notification outbox fields.
- Both migrations run transactionally under the existing PostgreSQL advisory migration lock. Repeated and concurrent `Open` calls are idempotent.
- The `2.4.19` migration has a distinct marker, so a database containing only the `2.4.18` marker still creates all broadcast delivery tables and columns.
- No watcher protocol, scheduler, chain-source, or watcher database behavior changes. Existing production watchers do not need to restart.
